# Архитектура: фьюжн и математика триангуляции

## 1. Поток данных

```
                 ┌────────────── главный поток (CPU, ~30-60 Гц) ─────────────┐
   экран ──grab──► minimap.read ─┐                                            │
        │                        ├─► fusion.step_player (Kalman) ─► снапшот ──┼─► navigator
        └──► optical_flow.track ─┘            ▲                               │   (state machine)
                    │  conf/residual          │ объекты в мире                │
                    ▼                         │                               │
              keyframe.Trigger ── fire? ──────┼───────────────┐               │
                                              │               ▼               │
              ┌── поток перцепции (GPU/iGPU, по событию) ──────────────────┐  │
              │  sam.segment(ROI) ─► semantic.classify ─► triangulator ────┼──┘
              │                                   └─► reseed optical_flow   │
              └────────────────────────────────────────────────────────────┘
```

Главный цикл никогда не блокируется на SAM: тяжёлый инференс уходит в отдельный поток и **вливается в fusion асинхронно**.

## 2. Когда вызывается SAM (событийный триггер)

Не по таймеру. Три условия (`perception/keyframe.py`):

1. **Покачивание камеры** (`on_camera_wobble`). Навигатор периодически качает камеру на известный угол для триангуляции — именно в этот момент и так нужна свежая точная маска. Совмещаем оба действия → не тратим лишний вызов GPU.
2. **Падение уверенности трекера** (`on_low_confidence`). Optical flow отдаёт число удержанных точек и residual (ожидаемая vs фактическая позиция маски). Мало точек или большой residual → дрейф накопился → ресинк, даже если камера не двигалась.
3. **Форс-интервал** (`max_interval_frames`). Верхний предел, чтобы дрейф в принципе не сломал математику триангуляции.

Чисто временной триггер плох: слишком часто — контеншн не уходит; слишком редко — дрейф ломает триангуляцию. Событийный попадает точно туда, где маска реально нужна.

## 3. Триангуляция: почему работает поворот камеры

Чистый поворот камеры вокруг **своей оптической оси** параллакса не даёт — все объекты (близкие и далёкие) смещаются на один и тот же угол. Но в Genshin камера от 3-го лица **орбитальная**: вращается вокруг персонажа на радиусе `R`. Поворот по yaw на `Δyaw` двигает оптический центр по дуге:

```
база  b = 2 · R · sin(Δyaw / 2)
```

Ненулевая база ⇒ параллакс ⇒ глубину можно вычислить. Дальше — обычная two-view триангуляция: два пеленга пикселя `u1, u2` из двух известных поз камеры дают пересечение лучей в мировых координатах (`triangulate_two_view`).

Те же формулы работают для:
- **шага/бега** — база = пройденный путь (скорость бега × время),
- **прыжка** — база = известная дальность прыжка.

Во всех случаях смещение камеры боту известно заранее → глубина объекта вычислима.

### Ожидаемое vs фактическое смещение (шаг 4 ТЗ)

Зная позу до/после и гипотезу позиции объекта, считаем **ожидаемый** пиксельный сдвиг (`expected_shift_from_motion`). SAM/optical-flow дают **фактический**. Разница:

- ≈ 0 — статичный объект на плоскости движения;
- систематическая — объект ближе/дальше/выше гипотезы (уточняем глубину);
- большая/хаотичная — объект движется сам (**монстр**).

## 4. Фьюжн миникарты и триангуляции (главный вопрос ТЗ)

### Позиция персонажа — комплементарный Kalman
Состояние `[x, y, vx, vy]` (`fusion/estimator.py::PoseKalman`):

- **predict** по скорости от optical-flow/миникарты (высокая частота, малая дисперсия `flow_vel_var`) — гладко, но копит дрейф;
- **correct** абсолютной позицией с миникарты (низкая частота, большая дисперсия `minimap_pos_var`) — убирает дрейф, но «дёргается».

Kalman автоматически взвешивает источники по их дисперсиям: между обновлениями миникарты траектория ведётся точной скоростью, а приход абсолютной позиции мягко стягивает оценку к истине. Итог — **гладкая и несмещённая** траектория.

### Позиция объекта — из камеры в мир + взвешенное слияние
1. Триангуляция даёт объект относительно персонажа (система камеры).
2. Берём отфильтрованную мировую позу персонажа и переносим объект в мир:
   `world_obj = player_world + fwd·z + right·x`.
3. Один объект, увиденный с разных поз, сливается как две гауссианы (information filter, веса = 1/дисперсия). Дисперсия измерения растёт с квадратом глубины (дальние объекты — менее точны).

Так **миникарта задаёт глобальный якорь координат**, а **триангуляция SAM уточняет и персонажа, и объекты** внутри этого якоря. Карта объектов (`WorldObject`) — то, по чему навигатор строит коридор и решает про обход.

## 5. Контеншн за GPU — три уровня смягчения

1. **Снижение частоты** (событийный триггер) — меньше суммарное время конфликта, но микростаттер в момент вызова остаётся: WDDM не делает application-aware приоритизацию между DirectX-игрой и CUDA-процессом, только тайм-слайсинг.
2. **ROI-кроп** — сегментируем только предсказанную область, а не весь кадр. Резко короче сам GPU-стоппер и меньше compute.
3. **OpenVINO на CPU/iGPU/NPU** — корневое решение: физически разводит инференс и рендер по разным устройствам, дискретка целиком остаётся игре. Если в системе есть iGPU — контеншн исчезает, а не смягчается.

`Sam3Tracker` (decoupled detector–tracker, Object Multiplex в 3.1) дешевле полного детектора, но всё равно на GPU — он решает «не гонять детектор каждый кадр», а не «не делить GPU с игрой». Поэтому межкадровый трекинг держим на CPU optical flow; Sam3Tracker подключаем только если LK не хватает по точности — и тоже через OpenVINO на CPU/iGPU.

## 6. Производительность: что выбрать под железо

| Конфигурация | Рекомендация |
|--------------|--------------|
| NVIDIA dGPU + есть iGPU | SAM/семантику → **OpenVINO на iGPU**, игра держит dGPU. Контеншн снят в корне. |
| Только NVIDIA dGPU (без iGPU) | SAM на dGPU, но: событийный триггер + ROI-кроп + downscale. Опц. **DeepStream** для эффективного видео-пайплайна. |
| Jetson / NVIDIA edge | **jetson-inference (SegNet)** — оптимизированный инференс под Tegra/любые NVIDIA. |
| Слабый/любой GPU, нет CUDA | **OpenVINO на CPU** + агрессивный ROI и редкие кейфреймы; основная работа на optical-flow. |
| AMD/Intel GPU | **OpenVINO** (vendor-agnostic) — единственный путь без CUDA. |

CPU-слой (миникарта + optical flow + Kalman + управление) на обычном игровом ПК спокойно идёт 30-60 Гц и **не трогает GPU вообще** — именно он несёт реальное управление, а SAM лишь периодически ресинкает.
